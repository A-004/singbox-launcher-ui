# SPEC 064 — Миграция UI с Fyne на Wails (HTML+CSS+JS)

> **Статус:** Черновик (planning)
> **Тип:** R — Refactor/Rewrite
> **Масштаб:** Migration (M)
> **Связанные:** CONSTITUTION.md, ARCHITECTURE.md, все SPEC-ы, затрагивающие UI

---

## 1. Проблема

Текущий UI написан на **Fyne** (Go-фреймворк с OpenGL-рендерингом). Это накладывает ограничения:

1. **Внешний вид** — нельзя использовать CSS, анимации, градиенты, кастомные шрифты, тени.
2. **Сборка** — требуется OpenGL и C-компилятор (MSYS2/MinGW на Windows).
3. **Гибкость** — сложно делать сложные layout'ы, таблицы, деревья, кастомные контролы.
4. **Сообщество** — Fyne маленькое, готовых компонентов мало.
5. **Производительность UI** — Fyne рисует всё через OpenGL, что на Windows без GPU даёт тормоза.

При этом **бэкенд** (core/, api/, internal/, core/services/) написан отлично, с чистой архитектурой, слоями, EventBus. **Менять его не нужно.**

---

## 2. Цель

Заменить **слой UI (L5–L7 по ARCHITECTURE.md)** на стек **Wails + HTML+CSS+JS**:

| Слой | Было (Fyne) | Стало (Wails) |
|------|-------------|---------------|
| L7 — виджеты | `internal/fynewidget` | CSS-компоненты (Button, Table, Dialog) |
| L6 — вьюхи | `ui/*.go`, `ui/configurator/tabs/*`, `ui/traffic/*` | HTML-страницы + SPA-роутинг |
| L5 — презентация | `ui/configurator/presentation/*`, `ui/configurator/business/*` | Go-бэкенд + JS bindings |
| L4 — API | `api/` — остаётся | `api/` — остаётся |
| L3 — сервисы | `core/services/*`, `core/uiservice/*` | Частично рефактор (убрать Fyne-зависимости) |
| L2 — домен | `core/state/*`, `core/build/*`, `core/config/*` | Не меняется |
| L1 — утилиты | `internal/*` | Не меняется |
| L0 — платформа | `internal/platform/*` | Не меняется |

**Бэкенд на Go остаётся ядром.** Меняется только то, как UI рисуется.

## 3. Архитектура Wails

```
┌─────────────────────────────────────────────────────┐
│                  Wails Application                    │
│                                                      │
│  ┌─────────────────────────┐  ┌───────────────────┐  │
│  │      Go Backend          │  │     Frontend       │  │
│  │                          │  │   (WebView)        │  │
│  │  core/     api/          │◄─┤                    │  │
│  │  internal/ core/services │  │  React/Vue/Svelte  │  │
│  │                          │  │  + HTML + CSS      │  │
│  │  Wails runtime bindings  │──►                    │  │
│  └─────────────────────────┘  └───────────────────┘  │
│                                                      │
│  - Системный WebView (Edge/WebKit, не Electron)      │
│  - Трей-иконка через systray (Wails встроенный)       │
│  - Нативный заголовок окна или frameless              │
│  - Размер бинарника ~15 MB                           │
└─────────────────────────────────────────────────────┘
```

### 3.1 Как работает связка

1. **Go-бэкенд** экспортирует функции через `wails:export` — они становятся доступны из JS.
2. **Frontend (HTML+JS)** вызывает эти функции через `window.runtime.Call()`.
3. **Wails runtime** автоматически генерирует TypeScript-типы для всех экспортированных функций.
4. **Callback'и** от бэкенда (например, обновление статуса VPN) приходят через Eventsystem Wails (`EventsOn`/`EventsEmit`).

### 3.2 Что остаётся от Fyne-слоя

- `core/uiservice` — **выпиливается** (Fyne-специфичные callback'и заменяются на Wails Events).
- `internal/apptheme` — **выпиливается** (заменяется CSS).
- `internal/fynewidget` — **выпиливается**.
- `internal/dialogs` — **заменяется** на нативные HTML-диалоги или Wails-диалоги.
- `internal/locale` — **остаётся** (i18n используется на Go-стороне).
- `internal/debuglog` — **остаётся**.

## 4. Философия миграции (как делать)

### 4.1 Принципы

1. **Бэкенд не трогать.** Ни одна строка в `core/`, `api/`, `internal/` (кроме `internal/apptheme`, `internal/fynewidget`, `internal/dialogs`) не меняется без крайней необходимости.
2. **Постепенная миграция.** Не "выпилить всё за день", а экран за экраном.
3. **Обратная совместимость.** Пока не весь UI перенесён, Fyne и Wails работают параллельно (как сейчас с webui-прототипом).
4. **Feature parity.** Каждый перенесённый экран должен иметь все функции, что были в Fyne.
5. **CSS-first.** Весь внешний вид — через CSS. Никакой inline-стилизации в JS.
6. **Минимум зависимостей.** Без Electron, без лишних npm-пакетов. React/Vue — опционально, можно и на чистом HTML+JS.

### 4.2 Очерёдность

| Этап | Что делаем | Время |
|------|-----------|-------|
| 0 | Установка Wails CLI + инициализация Wails-проекта внутри текущего | 1 день |
| 1 | **Дашборд** (статус VPN, прокси-лист, переключение) — уже есть прототип, перенести в Wails | 2 дня |
| 2 | **Core Dashboard** (скачивание ядра, обновление, версии) | 2 дня |
| 3 | **Clash API Tab** (таблица прокси, пинг, переключение, трафик) | 3 дня |
| 4 | **Diagnostics** (логи, debug API, профилировщик) | 2 дня |
| 5 | **Configurator** (весь визард: источники, outbounds, правила, DNS) | 2 недели |
| 6 | **Traffic Profiler** (графики трафика) — Canvas/JS | 3 дня |
| 7 | **Settings** (настройки, язык, тема) | 2 дня |
| 8 | **Трей-иконка** (системный трей с меню) | 1 день |
| 9 | **Удаление Fyne-кода** (после полной проверки feature parity) | 1 день |
| 10 | **Тестирование + багфикс** | 1 неделя |

**Итого ориентировочно:** 3-4 недели полной миграции.

### 4.3 Стек фронтенда (рекомендация)

- **Без фреймворка (vanilla HTML+JS+CSS)** — для этапов 1-4, минимум зависимостей.
- **Petite-Vue** (6KB) или **Alpine.js** — для реактивности в configurator (этап 5).
- **CSS:** кастомный (темная тема Apple, как в прототипе).
- **Иконки:** SVG спрайты (встроены в HTML).
- **Сборка:** esbuild (лёгкий, быстрый, входит в Wails).

## 6. Критические моменты

### 6.1 Трей-иконка

Wails поддерживает systray через `github.com/wailsapp/wails/v2/pkg/options`. Нужно будет перенести логику из `core/tray_menu.go` (чистая Go-логика, не Fyne) и переписать создание меню.

### 6.2 Диалоги (ShowError, ShowConfirm и т.д.)

Fyne-диалоги (`internal/dialogs`) заменяются на:
- `wails runtime MessageBox` для простых сообщений
- Кастомные HTML-диалоги (модальные окна) для сложных (редактор outbound'ов)

### 6.3 Профилировщик трафика (ui/traffic)

Текущий Fyne-виджет рисует графики через Canvas. В HTML это будет **Canvas 2D API** или **Chart.js** (маленькая библиотека, 70KB).

### 6.4 Configurator (визард)

Самый большой кусок (~15k строк Fyne). Состоит из:
- Sources (подписки) — список + добавление/редактирование
- Outbounds — глобальные outbound'ы + presets
- Rules — правила маршрутизации
- DNS — DNS-правила + предустановки

В HTML это будет SPA с несколькими вкладками. Petite-Vue сделает его реактивным без лишнего веса.

### 6.5 Локализация

В текущей архитектуре `internal/locale` используется на Go-стороне. В Wails:
- Базовые тексты кнопок/лейблов — можно передавать через bindings
- Либо перейти на i18n на фронтенде (i18next)

## 7. Риски и mitigation

| Риск | Mitigation |
|------|-----------|
| Потеря функциональности при переносе | Feature parity checklist на каждый этап |
| Долгая миграция (3-4 недели) | Параллельная работа Fyne + Wails (как сейчас с webui) |
| WebView может вести себя по-разному на Windows/macOS/Linux | Использовать стандартные CSS-свойства, тестировать на всех платформах |
| Потеря трей-иконки на macOS | Wails v2 поддерживает системный трей |
| Проблемы с производительностью WebView | Системный WebView (Edge/WebKit) быстрее Electron |
